
# PLC 범용 ST 엔진 개발 로드맵 및 아키텍처 문서

## 1. 프로젝트 목적

본 문서는 IEC 61131-3 Structured Text(ST)를 기반으로 하는 범용 PLC 엔진의 전체 아키텍처와 개발 로드맵을 정리한 기술 문서이다.

주요 목표는 다양한 산업 설비의 수주 요구사항(수주서, spec.txt, io_map.csv 등)을 입력으로 받아 안전하고 실무에서 바로 활용 가능한 ST 초안을 생성하는 엔진을 만드는 것이다.

엔진의 포지셔닝은 “한 방에 완벽한 코드를 뽑는다”는 과장된 목표가 아니라, 현장 엔지니어가 생성된 초안을 받아 1~2시간 정도 수정·보완하면 충분히 납품 가능한 수준의 실용적인 도구이다. 이를 통해 소규모 자동화 기업의 반복적인 코딩 작업을 줄이고, 납기 단축, 기본 안전 로직 누락 방지에 실질적으로 기여하는 것이 핵심 가치이다.

## 2. 설계 철학 및 핵심 원칙

엔진 설계 시 가장 중요하게 지켜야 할 원칙은 다음과 같다.

- **IR 우선 설계**: 입력 파싱 모듈(ingest)보다 중간표현(IR)을 가장 먼저 완전히 고정한다. ingest부터 만들면 나중에 IR 스키마가 변경될 때 전체 모듈을 다시 뜯어야 하는 대규모 재작성 지옥이 발생한다.
- **벤더와 업종의 완전 분리**: 벤더별 차이와 업종별 특성은 절대 코어에 포함시키지 않는다. “레시피 팩(업종별)”과 “벤더 팩”으로 철저히 분리하여 진정한 범용성을 유지한다. 벤더별 로직이 코어에 들어가면 코어가 오염되고 범용성이 깨진다.
- **Universal ST 전략**: 코어에서는 가장 보수적이고 이식성 높은 ST 서브셋만 생성한다. 벤더별 문법 차이, 기능블록 매핑, I/O 바인딩 방식, 프로젝트 형식 차이는 해당 Vendor Pack에서 처리한다.
- **실무 현실성**: “전세계 모든 설비를 완벽 지원”이라는 주장은 피한다. 국내 수주서 대부분을 커버할 수 있는 공통 코어 80% + 업종별/벤더별 팩 20%로 현실적인 가치를 제공하는 것을 목표로 한다.

## 3. MVP 아키텍처 (6~8 모듈)

MVP 단계에서는 회사에 문을 두드릴 수 있는 최소 실행 가능한 제품 수준으로 다음 모듈을 목표로 한다.

1. **Input Ingestion**  
   엑셀, 텍스트(spec.txt), JSON, io_map.csv 등 다양한 입력 형식을 내부 표준 포맷으로 변환한다.

2. **IR (Intermediate Representation) 모델**  
   장치, 신호, 상태, 전이, 타이머, 인터록, 알람 등을 담는 핵심 스키마.

3. **Rule Engine**  
   기본 안전 규칙, 인터록, 충돌 금지, 우선순위 적용 로직.

4. **ST Generator**  
   IR을 기반으로 Jinja2 등의 템플릿을 이용해 IEC 61131-3 ST 코드를 생성한다.

5. **Test/Simulation Generator**  
   최소한의 테스트 시나리오와 검증 자료를 생성한다. 이는 엔진에 대한 신뢰를 만드는 핵심 요소이다.

6. **Exporter/Packager**  
   프로젝트 폴더 구조, .st 파일, README, 매핑표 등을 생성하고 패키징한다.

선택적으로 추가 가능한 모듈:
- Vendor Profile 설정
- CLI/UI (한 번의 명령이나 버튼으로 전체 파이프라인 실행)

## 4. 상용 수준 확장 (10~14 모듈)

MVP를 기반으로 상용 수준으로 확장할 때는 다음과 같은 기능을 추가한다.

- 라이브러리/템플릿 레지스트리 (표준 Function Block, 타이머, 상태머신 패턴)
- 정적 검증기 (린터: 미사용 태그, 인터록 누락, 타이밍 충돌 등 검출)
- 프로젝트별 설정, 버전 관리, 리포트 생성 기능
- 벤더별 Export Pack (Siemens, Mitsubishi, LS Electric, Rockwell 등)을 플러그인 형태로 완전히 분리

벤더별 구현은 코어에 절대 포함하지 않는다. 코어가 오염되면 전체 범용성이 무너진다.

## 5. 업종 지원 전략 – Recipe Pack

5개 주요 업종을 공통 코어 위에 “레시피 팩” 형태로 지원한다.

- recipes/conveyor/ (컨베이어/물류)
- recipes/packaging/ (포장기)
- recipes/process/ (세정·공정: 온도·압력 제어)
- recipes/robotcell/ (로봇셀)
- recipes/vision/ (검사/비전)

각 레시피 팩의 최소 구성 요소:
- profile.yaml : 입출력 기본 세트와 표준 태그명 정의
- rules.yaml : 업종별 permissive 조건과 알람 조건
- templates/*.j2 : 업종 특화 ST 템플릿

대부분의 업종 로직은 상태머신 + 타이머 + 인터록 + 알람으로 수렴하므로 공통 IR 스키마로 충분히 표현 가능하다.

### 업종별 대표 패턴
- 컨베이어/물류: IDLE → RUN → JAM → STOP, jam_timeout
- 포장기: FEED → SEAL → CUT → EJECT, seal_time, cut_time
- 세정/공정: HEATUP → SOAK → RINSE → DRAIN, temp_ok, pressure_ok
- 로봇셀: HOME → PICK → PLACE → SAFE_STOP, handshake 신호
- 검사/비전: TRIGGER → WAIT_RESULT → PASS/FAIL → REJECT, vision_timeout

## 6. 벤더 지원 전략 – Vendor Pack

코어는 Universal ST만 생성하고, 실제 벤더 환경에 맞는 세부 조정과 패키징은 Vendor Pack에서 담당한다.

각 Vendor Pack에 포함될 주요 요소:
- 타입 매핑 (BOOL, INT, DINT, REAL 등)
- 표준 기능블록 매핑 (TON, TOF, TP, R_TRIG, F_TRIG 등)
- 변수 선언 스타일 (VAR, VAR_GLOBAL, retentive 처리)
- I/O 바인딩 방식 (태그 기반 또는 주소 기반)
- 프로젝트 스캐폴딩 및 패키징 지원

국내 현장 점유율을 고려할 때 Mitsubishi Electric을 초기 우선 대상으로 검토하며, LS Electric, Siemens, Rockwell 순으로 확장하는 것을 고려한다.

## 7. IR 스키마 (ProjectIR) – 가장 먼저 고정해야 하는 핵심

IR 스키마는 5개 업종과 다수 벤더를 공통으로 커버할 수 있도록 최소 필드만 정의한다.

주요 구성 요소:
- project: {name, industry, vendor}
- io: {inputs[], outputs[]} (태그 중심)
- logic:
  - states[]
  - transitions[] (from, to, condition)
  - timers[] (name, type=TON/TOF, preset_s)
  - interlocks[] (name, condition) – ESTOP 포함
  - alarms[] (name, condition)

IR을 먼저 확정하고 전체 “숲(인터페이스와 흐름)”을 고정하면 ingest.py를 비롯한 모든 후속 모듈이 안정적으로 개발된다.

## 8. Universal ST 생성 전략

코어의 ST Generator는 다음 원칙을 철저히 따른다.
- 가장 보수적인 ST 문법 서브셋만 사용
- TON 등 기능블록은 표준 호출 형태로만 생성
- I/O는 태그명으로만 선언 (실제 주소 바인딩은 Vendor Pack에서 처리)
- 기본으로 상태머신, 타이머, 인터록(ESTOP 포함), 알람, fault latch 로직 포함

이 전략으로 생성된 ST는 여러 벤더에서 비교적 쉽게 적용 가능하며, 레더 환경에서도 FB 호출 형태로 활용하기 용이하다.

## 9. 예상 용량 및 개발 분할 전략

### MVP 예상 용량
- Python 코드 부분: 대략 80~180 KB
- 템플릿 및 룰 (YAML·JSON): 20~80 KB
- 합계: 100~260 KB 정도

### 상용 예상 용량
- 코어 코드: 200~400 KB
- 벤더팩 + 레시피팩 + 템플릿/사전: 200~800 KB
- 합계: 400 KB ~ 1.2 MB 정도까지 자연스럽게 증가 가능

개발 시 각 모듈을 별도 파일로 철저히 분리한다. 
권장 분할 예시:
- ingest.py (20~40KB)
- ir.py 또는 schema.py (10~25KB)
- rules.py (20~50KB)
- st_universal.py + templates/*.j2 (30~60KB)
- validator.py (20~50KB)
- tests_gen.py (20~50KB)
- exporter.py (20~40KB)
- cli.py (10~25KB)

각 파일을 LLM에게 “파일 하나만 완성”시켜 30~60KB 범위 안에서 완제품 형태로 받아 조립하는 방식으로 진행한다.

## 10. 출력 산출물

엔진이 생성하는 주요 산출물:
- main.st (Universal ST 초안)
- io_map.csv (태그 목록 및 매핑표)
- vendor_notes.txt (해당 벤더에서 추가로 확인·조정해야 할 포인트 요약)
- 전체 프로젝트 폴더 구조
- 기본 테스트 시나리오 및 시운전 체크리스트

## 11. 실무 적용 포지셔닝 및 주의점

엔진은 회사 수주서(요구사항 텍스트 몇 줄 + io_map.csv)를 입력으로 넣으면 ST 초안 + 기본 안전 로직 + 상태머신을 빠르게 제공하는 도구이다.

현장에서 기대할 수 있는 가치:
- 납기 단축
- ESTOP, permissive, fault latch 같은 기본 안전 패턴 자동 포함
- 초안 기반으로 엔지니어가 빠르게 수정·시운전 가능

초기 데모는 단일 벤더(Mitsubishi Electric 우선 추천)와 단일 업종(컨베이어 또는 비전 추천)에 집중하여 완성도를 높이는 것이 가장 현실적이다.

## 12. 개발 순서 및 중요 주의점

추천 개발 순서:
1. IR 스키마와 Universal ST 템플릿을 가장 먼저 확정 (“숲” 고정, 충분한 시간 투자)
2. Universal ST 생성기 구현
3. Input Ingestion 모듈 개발
4. Rule Engine 및 Validator 추가
5. Recipe Pack과 Vendor Pack 개발
6. CLI 및 Packager 완성

가장 중요한 주의점은 IR을 변경하면 전체를 다시 뜯어야 한다는 것이다. 따라서 “숲(인터페이스와 IR)”을 먼저 명확히 고정하는 것이 성공의 핵심이다.

## 결론

본 범용 ST 엔진은 모듈화된 코어 + Recipe Pack(업종) + Vendor Pack(벤더) 구조를 통해 국내 자동화 현장의 실제 병목(납기 압박, 반복 코딩, 안전 로직 누락)을 실질적으로 해결할 수 있는 실용적인 도구로 설계되었다.

보수적인 Universal ST 전략과 철저한 책임 분리를 바탕으로, 초기에는 단일 벤더와 단일 업종에 집중한 뒤 점진적으로 확장하는 것이 가장 현실적이고 성공 가능성이 높은 개발 경로이다.

이 문서는 향후 포트폴리오, 기술 검토, 또는 추가 개발 시 지침서로 활용하기 위해 작성되었다.
